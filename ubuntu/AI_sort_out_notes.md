# 📝 项目笔记整理规范与 Prompt 模板

长对话排障后，若用 AI 辅助整理部署笔记，Prompt 宜包含 **「强制回溯 + 结构约束 + 细节定调」** 三要素。

## 💡 黄金提示词模板 (可以直接复制填空)

以下为可用的 Prompt 模板（按需修改项目名与路径）：

> **【角色设定】**
> 请以资深机器人工程师视角，将上述对话整理为面向开发者的 Markdown 部署笔记，步骤可照抄执行，技术细节尽量完整。

> **【强制回溯范围】**
> 请覆盖对话全程，勿只总结最近几轮。
> **【内容 Checklist（不可遗漏项）】**
> 笔记须包含以下板块：
> 1. **前置环境与跳板**：（写明最开始的特殊操作，如 Docker 提取、特定的版本号）。
> 2. **完整终端指令**：（所有的 `git clone`、`conda`、`export` 等命令必须用代码块完整写出，不可简略）。
> 3. **致命踩坑与解决方案**：（详细记录对话中出现的报错，比如 [报错A] 和 [报错B]，以及对应的解决指令和思路）。
> 4. **核心代码修改**：（精确到文件路径、行数和修改内容，不要用“修改某文件”这种模糊描述）。
> 5. **性能参数与极限测试**：（记录不同参数下的显存、内存、速度对比，给出最佳实践方案）。
> 
> 
> **【格式与语气约束】**
> * 使用清晰的 Markdown 结构（多用 `##`、`###`、`-` 和 `> 注意`）。
> * 语气要专业、硬核，带有排雷的实战感。
> * **绝对禁止**使用“省略号”或“等”来跳过步骤。如果指令很长，全部写出来。
> 
>

## 🔍 为什么这个模板能“治”好 AI 的毛病？

### 1. 锚定起点 (克服“聚光灯效应”)

AI 默认优先关注最近的 5-10 轮对话。通过在提示词中明确写出“从最开始的 [具体事件] 开始”（比如点名“Docker”），可在提示词中指定从某事件起回顾重新扫描对话历史的最顶端，把被降权的早期记忆强行捞回来。

### 2. 给出 Checklist (克服“语义压缩”)

若仅要求“写详细笔记”，模型可能省略长命令输出。应给出 1、2、3、4、5 的硬性列表，避免省略关键终端指令与报错记录。

### 3. 设定输出精度 (禁止“偷工减料”)

要求“精确到文件路径”、“禁止使用省略号”，是在切断 AI 生成长文本时的“偷懒路径”。

## 🚀 高阶技巧：两步走战略 (The "Outline First" Strategy)

长周期工程即使使用上述模板，即使是用上面的终极模板，AI 也可能因为一次性生成的字数限制（Token Limit）而中途断掉，或者依然漏掉细节。

这时候，最稳妥的做法是“先对大纲，再写正文”。

**第一步：索要大纲**

> “我要写一篇非常详细的工程文档，请仔细回顾我们关于这个机器人项目的所有对话，先给我列一个**详细的笔记大纲（精确到四级标题）**。里面要体现出所有的环境步骤、踩过的坑和最终配置。先不要写正文，确认大纲后再写正文。”

**第二步：查漏补缺**

> （若大纲遗漏步骤时补充）
> “大纲里漏了关于 `unset DISPLAY` 解决显卡死锁的那一步，把它加到第 3 节里。现在大纲完美了，请按照这个大纲，以极其详细的技术文档标准，把正文完整写出来。”
