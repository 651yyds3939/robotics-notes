# 多传感器融合 (Sensor Fusion)

> **核心定位**：单一传感器不可靠
> 👉 相关：[状态估计专题](./state_estimation.md) · [SLAM 专题](./slam.md) · [相机标定](./camera_calibration.md)——IMU 漂移、GPS 不进屋、相机怕暗光。传感器融合的核心是**让多个有互补特性的传感器互相纠错**，给出比任何一个单传感器都更准、更稳的状态估计。

---

## 第 0 章：传感器融合一句话

> **大白话**：IMU 高频但会漂（像闭眼走路，数步数走一会儿就歪了）。相机低频但不漂（像睁眼看路标纠正方向）。融合就是"每走一步用 IMU 猜位置，每看到路标用相机纠正猜测"。两个配合，既快又准。

---

## 第 1 章：为什么需要融合——传感器互补表

| 传感器 | 频率 | 优势 | 劣势 |
|--------|------|------|------|
| **IMU** | 100-500Hz | 高频，短时精度高 | 零偏漂移，积分误差累积（二次增长） |
| **视觉 (VIO)** | 10-30Hz | 无漂移（绝对参考） | 暗光/动态模糊失效，计算重 |
| **关节编码器** | 1kHz | 极高频，无漂移 | 只能给相对运动，触地才能推算基座 |
| **GPS/RTK** | 1-10Hz | 绝对位置，无漂移（室外） | 室内用不了 |
| **LiDAR** | 10-20Hz | 高精度距离，全天候 | 重、贵、功耗大 |

## 第 2 章：融合方法

### 2.1 互补滤波 (Complementary Filter)

最简单实用的方法——利用两个传感器在**频域上的互补特性**：

$$\hat{x} = \alpha \cdot x_{\text{high-freq}} + (1-\alpha) \cdot x_{\text{low-drift}}$$

> 经典案例：IMU 提供高频姿态变化（角速度积分），加速度计提供低频绝对参考（重力方向）。互补滤波器：快变信号信 IMU，慢变信号信加速度计。

### 2.2 EKF 多传感器融合

EKF 自然地支持多传感器：每个传感器对应一个**观测方程** $z = h(x)$。

```text
预测步：IMU 角速度+加速度 → 预积分 → 预测下一时刻状态
更新步：按传感器触发时序依次更新
  - 关节编码器触地 → 腿部里程计更新
  - 相机关键帧   → VIO 视觉更新
  - GPS 到达     → 全局位置更新
```

> **关键**：EKF 的 $R$ 矩阵（测量噪声）决定了每个传感器的"话语权"。$R$ 小 = 这个传感器准，EDF 会给它更高的更新权重。

## 第 3 章：人形机器人的传感器融合架构

```text
IMU (200Hz) ──────────┐
                       ├──→ EKF → 基座 6-DOF 位姿+速度
编码器 (1kHz) ────────┤           → IMU 零偏估计
    ↓ 正运动学         │           → 接触状态（每腿 bool）
腿部里程计             │
                       │
相机 VIO (15Hz) ──────┘
```

### 关键工程实践

- **时间戳对齐**：所有传感器必须用同一个时钟源（PTP/NTP），毫秒级的时偏会导致 EKF 发散
- **异常值拒绝**：用马氏距离（Mahalanobis Distance）检测异常观测，拒绝后不做更新
- **触地感知**：只有踩在地上的腿才参与融合，摆动腿的运动学不准确

## 关键词速查

| 术语 | 解释 |
|------|------|
| **Complementary Filter** | 频域互补滤波器，低频和高频传感器互相填坑 |
| **IMU Preintegration** | IMU 预积分：两帧之间 IMU 测量的累积，避免重复计算 |
| **Mahalanobis Distance** | 归一化误差，用于检测"这个观测是不是离谱" |
| **Temporal Alignment** | 时间戳对齐：确保 EKF 融合的是同一时刻的数据 |
| **Outlier Rejection** | 异常值拒绝：马氏距离超过阈值的观测不参与更新 |
