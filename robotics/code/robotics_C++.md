## 🗺️ 模块一：核心技能地图（你要打哪几场仗？）

不要用“互联网纯软件开发”的思路去学机器人的 C++。这里的核心是**实时性**、**空间感知**与**数学计算**。

### 1. C++ 四大战区

| 核心战区 | 重点掌握内容 | 机器人实战场景 |
| --- | --- | --- |
| **内存与生命周期** | **RAII 机制、智能指针** (`shared_ptr`, `unique_ptr`)、值传递 vs 引用传递 `&` | 高频传感器数据流转，避免内存泄漏导致节点崩溃。 |

| **工程化工具** | **CMake** (`CMakeLists.txt`), 编译链接机制 | 把 Eigen、OpenCV、ROS 包链接到你的代码里。 |

## 🧠 模块二：认知跃迁 —— 脚本思维 vs 编译思维

从 Python 转向 C++，首先要适应底层机器视角。

| 维度 | Python (同声传译 / 动态化) | C++ (出版成书 / 静态化) |
| --- | --- | --- |
| **运行机制** | **解释执行**：走到错误的那一行才会崩溃。 | **编译执行**：必须经过编译器严格的安全检查，打包成二进制才允许运行。 |
| **内存模型** | **便利贴**：变量名随时可以贴给不同的数据对象。 | **定制铁盒**：强类型，声明后内存大小和形状永久固定。 |
| **垃圾回收** | **GC 机制**：后台打扫垃圾，**会引发不可控的全局停顿（卡顿）**。 | **RAII 机制**：靠作用域和智能指针实现**无延迟的精确释放**。 |

> ⚠️ **高危实战坑点：隐式截断**
> 将小数（如 `2.8`）强塞入整数 `int` 变量中，C++ 不会报错，而是会**无声无息地截断为 `2**`。这在控制算法中极易导致 PID 参数失效。

## 📂 模块三：物理架构 —— `.h` 与 `.cpp` 的分离哲学

工业级项目中，声明和实现是**严格分离**的。

| 文件类型 | 餐厅类比 | 核心作用 | 包含什么代码？ |
| --- | --- | --- | --- |
| **`.h` / `.hpp` (头文件)** | **菜单 (Menu)** | 告诉其他模块“我这里有什么接口”。 | 类的定义、变量声明、方法声明（带有返回类型、参数，但**没有**具体逻辑）。 |
| **`.cpp` (源文件)** | **后厨 (Kitchen)** | 真正干活的地方，实现承诺的功能。 | 包含 `#include "xxx.h"`，并写满 `类名::函数名 { 具体逻辑 }`。 |

> **💡 读码路径建议：**
> 看别人写的 ROS Node，**第一步永远是打开 `.h` 文件**，看它订阅了什么话题、发布了什么话题。看懂了“菜单”，再去 `.cpp` 里看“炒菜过程”。

## ⚡ 模块四：极速与安全 —— 内存与数据传递

在 MPC 等高频控制（100Hz+）中，严禁直接传递大型对象（如数万个点的轨迹）。必须通过地址传递实现**零拷贝**。

### 1. 黄金搭档：`const Type&` (常量引用)

* **& (引用)**：给原数据起“别名”，直接操作物理内存，**零拷贝**。
* **const (只读锁)**：承诺绝对不篡改原数据。
* **应用场景**：源码函数参数中最常见（如 `void updateMap(const PointCloud& cloud)`），性能极速且绝对安全。

### 2. 现代内存管理：智能指针 (Smart Pointers)

全面废弃手动 `new/delete`。

* **`std::shared_ptr` (共享指针)**：ROS 绝对核心。内部有**引用计数器**，最后一个使用者释放时，内存瞬间自动回收。永远用 `std::make_shared<T>()` 创建。
* **`std::unique_ptr` (独占指针)**：同一时刻只能有一个所有者，适合移交数据所有权。
* **源码缩写 `ConstPtr**`：全拼为 `std::shared_ptr<const MessageType>`。意思是“共享的、只读的智能指针”，常用于 ROS 传感器回调函数。

## 🧩 模块五：接口与语法糖 —— 现代 C++ 识谱

### 1. 面向对象与多态 (Polymorphism)

控制器通常需要适配不同的物理模型，靠“多态”解耦：

* **定义插座 (基类)**：用 `virtual` 声明虚函数（或 `= 0` 纯虚函数）。
* **制造电器 (派生类)**：用 `override` 明确重写具体逻辑。
* **动态绑定**：通过**父类指针**调用，运行时会自动执行子类的代码。

### 2. 必须看懂的语法糖

* **`auto`**：**编译期**自动推断类型。遍历修改大型容器必须加引用 `auto&`，只读遍历用 `const auto&`。
* **范围 For 循环**：`for (const auto& point : path) { ... }` (极具 Python 风格)。
* **Lambda 表达式**：`auto my_func = [](double a) { return a * 2; };` (常用于 ROS 定时器或并发控制)。

## 💡 模块六：终极口诀 —— 冒号家族解码器

底层源码中冒号只有 6 个高频场景，请建立“视觉反射”：

### 第一家族：双冒号 `::` (认祖归宗)

**作用域解析符**（XX 里面的 XX）。例如：`ros::Time::now()`。

### 第二家族：单冒号 `:` (身兼五职看位置)

1. **在 `class` 第一行？ ➡️ 继承** (`class Lidar : public Sensor`)
2. **在 `public/private` 后面？ ➡️ 门禁区域开始**
3. **在 `for` 循环括号里？ ➡️ in (遍历)**
4. **在构造函数名和 `{` 中间？ ➡️ 出厂设置 (初始化列表)**
* `MyNode() : speed(0) {}`（在节点创建瞬间，立刻把 speed 设为 0。ROS极高频）。


5. **旁边有个 `?` 吗？ ➡️ 否则 (三元运算符)**

## 🏭 模块八：构建引擎 —— CMake 最小生存指南

代码写完必须靠 CMake 才能“拼装成车”。看懂 `CMakeLists.txt` 的 5 个核心步骤：

```cmake
# 1. 声明项目名称
project(my_robot_controller)

# 2. 找帮手（寻找 ROS 库、Eigen 矩阵库等外部依赖）
find_package(catkin REQUIRED COMPONENTS roscpp nav_msgs)
find_package(Eigen3 REQUIRED)

# 3. 告诉编译器，头文件（菜单）在哪里
include_directories(
  include
  ${catkin_INCLUDE_DIRS}
  ${EIGEN3_INCLUDE_DIR}
)

# 4. 把 .cpp 源文件打包成一个可执行的 ROS 节点程序（造车）
add_executable(mpc_node src/mpc_node.cpp src/circle_path_generator.cpp)

# 5. 把找来的帮手和造好的车绑定在一起（上牌照）
target_link_libraries(mpc_node ${catkin_LIBRARIES})

```

## ☠️ 模块九：崩溃自救 —— 段错误 (Segfault) 排雷指南

Python 报错会告诉你行号，而 C++ 运行时崩溃通常只留下一句冰冷的：`Segmentation fault (core dumped)`。

遇到段错误，立刻排查以下**三大作死行为**：

1. **访问了空指针 (Null Pointer)**
* *场景：* 声明了雷达数据的指针，但**还没收到数据（未初始化）**，就直接用 `->` 去读坐标。
* *自救：* 使用指针前，永远加一句判断 `if (lidar_ptr == nullptr) return;`。


2. **数组越界 (Out of Bounds)**
* *场景：* `vector` 里只有 10 个点，算法强行读取第 11 个点 `path[10]`。
* *自救：* 永远使用 `.at()` 替代 `[]`（如 `path.at(10)`），这样越界时会抛出带行号的异常，而不是直接闪退。


3. **迭代器失效**
* *场景：* 在 `for` 循环遍历一个 `vector` 时，又在循环内部往这个 `vector` 里 `push_back` 塞新数据，导致底层内存地址集体大搬家，循环指针瞬间迷路崩溃。
